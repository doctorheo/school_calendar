# 사용자 입력 Config 파일 전환 구현 계획

## 목표

현재 프로젝트는 실행 시 커맨드라인 인자와 일부 환경변수로 입력값을 받는다. 이를 설정 파일 기반으로 전환해 실행 시 필요한 값을 한 곳에서 관리할 수 있도록 한다.

## 구현 항목

- 설정 파일 포맷 선정 및 기본 예시 파일 추가
- 설정 파일을 읽어 애플리케이션 입력값으로 변환하는 로더 추가
- `src/main.py`의 입력 처리 흐름을 config 기반으로 전환
- 필수값, 충돌 조건, 기본값에 대한 검증 로직 정리
- README 및 실행 예시를 config 파일 기준으로 갱신

## 구현 순서

1. 설정 파일 포맷을 결정한다.
2. 프로젝트 루트에 예시 설정 파일을 추가하고, 현재 `--url`, `--parsed-output`, `--calendar-id`, `--create-calendar`, `--calendar-name`, `--calendar-description`, `--timezone`, `--credentials-path`를 대응시키는 키 구조를 설계한다.
3. `src/main.py` 또는 별도 모듈에 config 로더를 추가해 파일을 읽고 기본값과 사용자 설정을 병합한다.
4. 기존 `argparse` 중심 흐름을 정리하고, 최소한의 진입 인자만 남기거나 완전히 config 파일 입력으로 전환한다.
5. `create_calendar`와 `calendar_id`의 동시 사용 금지, 새 캘린더 생성 시 이름 필요 여부, 파일 경로 유효성 등 설정 검증을 추가한다.
6. `.env`와 config 파일의 역할을 분리한다.
7. `README.md`, `.env.example`, 실행 스크립트를 config 파일 기준 사용 방법으로 수정한다.
8. 샘플 config로 실행해 기존 캘린더 저장 흐름과 새 캘린더 생성 흐름이 모두 동작하는지 수동 검증한다.

## 설계 기준

- Google 서비스 계정 파일 경로처럼 민감하지 않은 실행 입력은 config 파일로 이동한다.
- 비밀값 자체를 직접 저장해야 하는 요구가 없다면 인증 비밀은 계속 `.env` 또는 외부 파일 경로 참조로 유지한다.
- config 파일이 없거나 필수 키가 빠진 경우에는 즉시 이해하기 쉬운 에러를 반환한다.
- 기존 사용자가 크게 혼란스럽지 않도록 필요하면 한동안 CLI와 config를 함께 지원하되, 우선순위를 명확히 정한다.

## 검증 포인트

- config 파일만으로 프로그램이 실행되는지 확인
- 기존 `calendar_id` 기반 저장 흐름이 유지되는지 확인
- 새 캘린더 생성 흐름에서 이름과 설명이 올바르게 반영되는지 확인
- 충돌 설정이나 누락 설정에서 에러 메시지가 명확한지 확인
- 문서 예시와 실제 실행 방식이 일치하는지 확인
