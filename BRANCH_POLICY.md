# 브랜치 운영 규칙

## 목적
- 브랜치 역할을 명확히 구분하여 안정적인 배포와 협업을 유지한다.

## 브랜치 종류
- `main`: 운영 및 배포 기준 브랜치
- `dev`: 기능 통합 및 검증 브랜치
- `feat/agent-name/feature-name`: 기능 개발 브랜치
- `fix/agent-name/issue-name`: 버그 수정 브랜치
- `etc/task-name`: 기타 작업(문서/설정/운영성 변경) 브랜치

## 기본 작업 순서
1. `dev` 최신 변경사항을 반영한다.
2. 작업 시작 전에 목적에 맞는 새 브랜치를 생성한다.
3. 코드 수정 및 테스트를 수행한다.
4. 변경 파일을 스테이징(`git add`)하고 커밋한다.
5. 원격 브랜치로 푸시 후 `dev` 대상으로 PR을 생성한다.

## 머지 기준
- `feat/*`, `fix/*` 브랜치는 `dev`로 머지한다.
- 릴리스 시점에 `dev`를 `main`으로 머지한다.
- 긴급 수정이 `main`에 반영된 경우 `dev`에도 동일 변경을 반영한다.

## 네이밍 규칙
- 브랜치 이름은 소문자와 하이픈(`-`)을 사용한다.
- 작업 목적이 드러나도록 `feature-name`, `issue-name`을 명확히 작성한다.
